## 一、当前工作流的问题分析

### 1.1 一次生成模式的结构性缺陷

当前采用的工作流是：**一次性读取所有源文件 → 压缩摘要 → 一次性生成全文 → 事后检查**。

在本次7RP PCR临床试验方案的撰写实践中，暴露了以下典型问题：


| 问题类型          | 具体表现                                                      | 根因                       |
| ------------- | --------------------------------------------------------- | ------------------------ |
| **逻辑链断裂**     | 用目标值法算出阳性≥239、阴性≥110，但在"综合确定样本量"段落未做取最大值的运算，直接写了"不少于XXX例" | 一次生成中上下文压缩导致前后逻辑脱节       |
| **参数照搬**      | P₀=90%、P_T=95%直接从定量模板抄来，未考虑7重试剂盒的实际场景和厂家控制成本的需求           | 没有行业经验输入，agent只能照搬模板     |
| **隐性规则缺失**    | 样本量论证应参考已上市竞品实际入组数、精确概率法可能有更优结果等——这些均未体现                  | 当前工作流只有显性规则输入通道，无隐性知识机制  |
| **定量/定性适配粗糙** | 需要人工审核才能确认定量专属内容是否全部删除、定性专属内容是否全部新增                       | 缺乏结构化的"删除清单/新增清单"和对应验证机制 |


### 1.2 核心矛盾

当前工作流试图用**一次prompt**解决所有问题，但临床试验方案的撰写本质上是**多知识源、多约束、需迭代验证**的工作。单次生成的上下文窗口压力和注意力衰减，注定无法可靠地维持长逻辑链。

## 二、优化方向：指令式模板 + 工具链 + 逐章执行

### 2.1 整体架构

```
┌─────────────────────────────────────────────────┐
│                 CLAUDE.md 主控指令               │
│         （全局工作流定义、角色约束、输出规范）       │
└─────────────────┬───────────────────────────────┘
                  │
┌─────────────────▼───────────────────────────────┐
│          章节指令模板库（.md格式）                  │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐             │
│  │第三章 │ │第五章 │ │第六章 │ │...   │             │
│  │背景资料│ │试验设计│ │统计学│ │      │             │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──────┘             │
└─────┼────────┼────────┼─────────────────────────┘
      │        │        │
┌─────▼────────▼────────▼─────────────────────────┐
│              MCP工具层                             │
│  ┌───────────────┐  ┌──────────────┐             │
│  │样本量计算器     │  │样本量逻辑验证器│             │
│  │(生成计算结果)   │  │(语义级验证)   │             │
│  └───────────────┘  └──────────────┘             │
│  ┌───────────────┐  ┌──────────────┐             │
│  │统计区间计算器   │  │竞品数据库查询  │             │
│  │(Wilson/Kappa)  │  │(NMPA官网搜索) │             │
│  └───────────────┘  └──────────────┘             │
│  ┌───────────────┐                               │
│  │法规知识库       │                               │
│  │(结构化约束)     │                               │
│  └───────────────┘                               │
└─────────────────────────────────────────────────┘
      │
┌─────▼────────────────────────────────────────────┐
│              Hook兜底层（仅文本级检查）              │
│  定量专属词残留 / XXX占位符 / 必选词缺失 /         │
│  LLM是否跳过了验证工具调用（执行纪律兜底）          │
└─────────────────────────────────────────────────┘
```

### 2.2 三层协作关系


| 层级         | 载体      | 职责                                           | 强制性                 |
| ---------- | ------- | -------------------------------------------- | ------------------- |
| **规划层**    | .md文件   | 定义工作流步骤、章节结构、写作规范、隐性经验、**规定验证工具调用步骤**        | 指导性（agent应遵循但可能偏差）  |
| **计算与验证层** | MCP工具   | 执行确定性计算（样本量、置信区间等）+ **语义级逻辑验证**（LLM以结构化参数调用） | 强制性（调用即返回精确结果）      |
| **兜底层**    | Hooks脚本 | **仅做文本级检查**（关键词匹配），以及检查LLM是否跳过了必选的验证工具调用     | 兜底性（主力是.md指令+MCP工具） |


**关键设计原则**：

- **语义级验证交给MCP工具**，由LLM以结构化参数调用（如 `validate_sample_size_logic(statistical_positive=239, final_positive=239, ...)`），不依赖脚本解析自然语言
- **文本级检查交给Hooks**，只做正则/关键词级别的检查（如"文件中是否出现Deming""是否有XXX占位符""是否包含混合感染"）
- **Hook是兜底机制，不是主力**。主力是.md章节指令中规定的验证步骤（"写完后必须调用验证工具"），Hook只防止LLM偷懒跳过

**不需要传统框架或硬编码pipeline**，Claude Code原生的CLAUDE.md + MCP + Hooks已能实现上述三层结构。

## 三、章节指令模板的设计规范

### 3.1 单章节指令结构

每个章节的指令模板应包含以下字段：

```markdown
## [章节编号] [章节标题]

### 数据源
- [来源类型]: [具体文件/工具/API]
- [来源类型]: [具体文件/工具/API]
...
（来源类型包括：模板文件、考核试剂说明书、对比试剂说明书、法规文件、
MCP工具、网络搜索、竞品数据库、内部经验库）

### 计算规则
- 工具: [MCP工具名称及参数说明]
- 硬性约束: [法规/标准明确要求的不可违反的条件]
- 柔性约束: [需要论证但可灵活调整的条件]
- 输出要求: [必须展示哪些中间步骤和最终结论]

### 隐性规则
- 行业惯例: [非法规明文规定但行业默认遵循的做法]
- 厂家偏好: [基于实际业务需求的经验性指导]
- 论证技巧: [有助于提高审评通过率的写作策略]

### 删除清单（仅限从定量模板适配定性时）
- [需删除的具体内容及其原因]

### 新增清单
- [需新增的具体内容及其法规依据]

### 验证检查项
- [ ] [检查项1]
- [ ] [检查项2]
...
```

### 3.2 示例：第六章 样本量估算

```markdown
## 六（一）1 样本量估算

### 数据源
- 法规: 呼吸道病毒多重核酸检测试剂注册技术审查指导原则(2019) → 第(七)节
- 法规: 新型冠状病毒核酸检测试剂注册审查指导原则(2023修订) → 第(四)节
- 法规: 流行性感冒病毒抗原检测试剂注册审查指导原则(2023修订) → 第(四)节
- 竞品调研: 搜索NMPA已上市同类产品（多重呼吸道病原体核酸试剂盒）的
  临床试验方案/报告中记载的实际入组数，作为样本量论证的参考依据

### 计算规则
- 工具: 调用 sample_size_calculator 工具
  - 参数: target_value(P₀), expected_value(P_T), alpha, beta, dropout_rate
  - 方法: 支持选 "exact"(Clopper-Pearson精确概率法) / "normal_approx"(正态近似法) / "wilson"(Wilson得分区间法)
  - 约束输入: regulatory_minimums = {per-pathogen的法规硬性最低要求}
  - 参考输入: competitor_samples = [竞品实际入组数列表]
- 硬性约束:
  - SARS-CoV-2 阳性 ≥ 200例（新冠法规）
  - SARS-CoV-2 阴性 ≥ 300例（新冠法规）
  - 弱阳性样本 ≥ 50例（新冠法规）
  - 其他每种病原体阳性 ≥ 50例（呼吸道多重法规）
  - ≥ 3家临床试验机构（三份法规一致）
- 柔性约束:
  - P₀和P_T的取值需提供理论依据，不限定具体数值
  - 可按病原体流行率分层，对不同病原体采用不同的P₀/P_T
  - 可选用精确概率法，若计算结果优于正态近似法则在论证中明确说明
- 输出要求:
  - 展示完整的计算过程和参数选择依据
  - 明确展示"统计估算值 vs 法规最低值 → 取最大值"的逻辑链
  - 最终数字必须是确定值（不得留XXX占位符）
  - 附竞品实际入组数作为参考论证

### 隐性规则
- 厂家倾向: 在满足法规硬性要求的前提下，尽量控制总样本量
  （样本量直接影响临床试验的成本、周期和操作难度）
- 经验参考: 国内已上市多重呼吸道病原体核酸类产品的实际入组数
  通常在XXX~XXX例范围，可搜索NMPA临床试验登记平台获取
- 论证技巧:
  - 精确概率法在相同置信度下通常比正态近似法得出更小的样本量
  - 对于低流行率病原体，可结合流行病学数据论证入组可行性
  - 引用竞品实际入组数作为"行业惯例"参考可增强论证说服力

### 验证检查项
- [ ] 最终样本量的阳性数 ≥ max(统计估算阳性数, 法规最低阳性数)
- [ ] 最终样本量的阴性数 ≥ max(统计估算阴性数, 法规最低阴性数)
- [ ] 总样本量 = 阳性数 + 阴性数，且包含脱落率加成
- [ ] 样本量估算段落中出现的每一个中间计算数字，
      都在"综合确定"段落中被明确引用或解释
- [ ] 无XXX占位符残留

### 执行步骤（必须严格按顺序）
1. 调用 sample_size_calculator 工具，传入参数获取计算结果
2. 根据计算结果撰写本节正文
3. **撰写完成后，必须调用 validate_sample_size_logic 工具**，
   将正文中确定的每个数值作为结构化参数传入，验证逻辑链完整性
4. 若验证工具返回不通过，根据提示修正正文，然后重新调用验证工具，直到通过
```

## 四、MCP工具层的封装

### 4.1 优先级排序


| 优先级  | 工具                 | 理由                                      |
| ---- | ------------------ | --------------------------------------- |
| 🔴 高 | **样本量计算器**         | 使用频率最高，且最容易因手算出错；现有R代码可直接封装             |
| 🔴 高 | **样本量逻辑验证器**       | 验证样本量章节中各数值的逻辑链自洽性（语义级，LLM以结构化参数调用）     |
| 🔴 高 | **置信区间计算器**        | Wilson得分区间、Clopper-Pearson精确区间等，方案中大量使用 |
| 🟡 中 | **竞品数据库/NMPA搜索**   | 背景资料和同类产品章节需要；可先手动收集，后期自动化              |
| 🟡 中 | **法规知识库查询**        | 法规条款的结构化查询，避免每次重新解析全文                   |
| 🟢 低 | **Kappa/一致性统计计算器** | 使用频率较低，方案阶段通常只写方法不实际计算                  |


### 4.2 样本量计算器的接口设计

```python
sample_size_calculator(
    # 必填参数
    target_value: float,         # P₀，目标值（临床可接受标准下限）
    expected_value: float,       # P_T，预期值
    alpha: float,                # 显著性水平（默认0.05）
    beta: float,                 # II类错误率（默认0.20）
    dropout_rate: float,         # 脱落率（默认0.05）
    
    # 方法选择
    method: str,                 # "exact" | "normal_approx" | "wilson"
    
    # 约束输入
    regulatory_minimums: dict,   # {"SARS-CoV-2_pos": 200, "SARS-CoV-2_neg": 300, ...}
    competitor_samples: list,     # [竞品A实际入组数, 竞品B实际入组数, ...]
    
    # 病原体信息（用于报告）
    pathogen_name: str,
    assay_type: str,             # "qualitative" | "quantitative"
)
```

**返回值应包含**：

- statistical_positive / statistical_negative（统计估算值）
- regulatory_positive / regulatory_negative（法规最低值）
- final_positive / final_negative（取max后的最终值）
- total_with_dropout（加脱落率后的总入组数）
- calculation_details（完整计算过程，可直接嵌入方案文本）
- recommendation（建议和注意事项，如"精确概率法比正态近似法少XX例"）

### 4.3 样本量逻辑验证器的接口设计

本工具与4.2的计算器是互补关系：计算器**生成**数字，验证器**检查**数字是否自洽。
验证器由LLM在写完样本量章节后主动调用，传入结构化参数（非自然语言），返回通过/不通过及具体原因。

```python
validate_sample_size_logic(
    # LLM从正文中提取的结构化数值
    statistical_positive: int,          # 统计估算阳性数（从正文中识别）
    statistical_negative: int,           # 统计估算阴性数
    regulatory_positive_min: dict,      # 法规最低阳性数 {"SARS-CoV-2": 200, ...}
    regulatory_negative_min: dict,      # 法规最低阴性数 {"SARS-CoV-2": 300, ...}
    final_positive: int,                # 正文中"综合确定"段落声明的最终阳性数
    final_negative: int,                # 正文中"综合确定"段落声明的最终阴性数
    final_total: int,                  # 正文中声明的总样本量
    dropout_rate: float,               # 脱落率（如果有）
)

# 返回值示例（通过）
{
  "passed": true,
  "details": [
    "final_positive(239) >= max(statistical_positive(239), regulatory_max(200)) ✓",
    "final_negative(300) >= max(statistical_negative(110), regulatory_max(300)) ✓",
    "final_total(539) >= (239+300)/(1-0.05) = 566.3 → 实际需≥567，当前539不足"
  ]
}

# 返回值示例（不通过）
{
  "passed": false,
  "errors": [
    "final_total(539) < 567（加脱落率后最低要求），需修正"
  ]
}
```

**设计要点**：LLM写完正文后，已经"知道"每个数字的含义，只需把这些数字作为结构化参数传给验证器。
验证器做纯算术比较，不解析自然语言。这样既避免了Hook脚本用正则切自然语言的脆弱性，又保证了验证的确定性。

### 4.4 工具封装的技术选型

现有R代码可直接通过MCP Server封装为工具，推荐方案：

- **方案A**：Python FastMCP（推荐，轻量级，与Claude Code的MCP集成无缝）
- **方案B**：直接用R的httpgd/plumber包暴露为API，Python层转发
- **方案C**：将R代码翻译为Python（scipy.stats有Clopper-Pearson等现成实现）

### 4.5 MCP工具的工作原理（供技术参考）

MCP（Model Context Protocol）对LLM完全透明。LLM只认识Anthropic API的三个原语：`tools`（工具定义）、`tool_use`（调用）、`tool_result`（结果）。

Claude Code客户端负责把MCP Server暴露的工具翻译成这三个原语：

```
1. Claude Code启动时，通过stdio连接MCP Server进程，获取所有工具的JSON Schema
2. 每次API请求时，将Schema塞入tools参数发给LLM
3. LLM决定调用某个工具时，返回tool_use block（含工具名+结构化参数）
4. Claude Code通过stdio将参数转发给MCP Server执行
5. MCP Server返回JSON结果，Claude Code组装为tool_result发回API
6. LLM读取结果，继续生成内容
```

因此，MCP Server的`input_schema`设计质量直接决定LLM能否正确调用。
描述清晰、参数有默认值、枚举值明确的Schema，能大幅降低传参错误率。

**注册方式**：

```bash
# 项目级
claude mcp add sample-size -- python3 path/to/server.py

# 用户级（所有项目可用）
claude mcp add --scope user sample-size -- python3 path/to/server.py
```

配置写入`.claude/settings.json`或`~/.claude/settings.json`，Claude Code下次启动时自动拉起MCP Server进程。

## 五、Hook兜底层

### 5.1 Hook的定位：兜底而非主力

.md指令告诉LLM"你应该做什么"，MCP工具提供验证能力，但**两者都不能保证LLM一定执行了验证步骤**——LLM可能跳过。

Hook的作用是在客户端侧提供**不可绕过的兜底检查**。但Hook的能力边界必须明确：


| 检查类型                     | 适合谁做      | 原因                          |
| ------------------------ | --------- | --------------------------- |
| **文本级检查**（关键词是否存在/不存在）   | Hook ✅    | 正则匹配即可，确定性高，不依赖语义理解         |
| **语义级验证**（数值逻辑链是否自洽）     | MCP验证工具 ✅ | 需要理解数字的含义和关系，LLM以结构化参数调用最可靠 |
| **执行纪律检查**（LLM是否跳过了必选步骤） | Hook ✅    | 检查工具调用记录即可                  |


**核心原则：Hook不解析自然语言。** LLM产出的是自然语言Markdown，用Python脚本的正则去切中文自然语言提取数值，极其脆弱（"阳性不少于239例"vs"阳性入组数确定为239例"vs"阳性≥max(239,200)=239例"，写法稍有变化就崩）。语义级验证的正确做法是让LLM以结构化参数调用MCP验证工具（见4.3节）。

### 5.2 Hook适合做的事：文本级检查

#### 检查点1：定性/定量适配完整性

**触发时机**：写入方案文件后

**检查逻辑**（纯关键词/正则匹配）：

```
定量专属内容删除检查（应全部不存在）:
  ✓ 正则匹配: /Deming|Passing-Bablok|Bland-Altman|偏差图|离群值/
  ✓ 正则匹配: /Pearson|Spearman|Kendall|相关性分析/
  ✓ 正则匹配: /医学决定水平.*偏倚/

定性专属内容新增检查（应全部存在）:
  ✓ 正则匹配: /混合感染/
  ✓ 正则匹配: /分层分析|年龄段/
  ✓ 正则匹配: /弱阳性/
  ✓ 正则匹配: /卡方检验|Kappa/
```

这些检查不依赖理解语义，只需确认某些关键词的出现/不出现，正则完全胜任。

#### 检查点2：占位符残留

**触发时机**：每次写入.md文件后

```
检查文件中是否包含 XXX 或 XXXX 占位符
→ 有则报错
```

#### 检查点3：执行纪律兜底（提醒LLM调用验证工具）

**触发时机**：写入样本量相关章节后

Hook可以用来检查LLM是否跳过了必选的MCP验证工具调用。这属于"执行纪律"检查，不需要解析文件内容：

```python
# 伪代码示意
import json, sys

# Claude Code通过stdin传入工具调用上下文
tool_info = json.load(sys.stdin)
file_path = tool_info.get("file_path", "")

# 仅在样本量相关文件中触发
if "样本量" not in file_path:
    sys.exit(0)

# 检查本轮对话中是否调用了validate_sample_size_logic工具
# （通过检查Claude Code的工具调用日志或对话历史）
has_validation_call = check_tool_call_history("validate_sample_size_logic")

if not has_validation_call:
    print("你刚写完了样本量估算章节，但尚未调用validate_sample_size_logic"
          "验证工具。请立即调用该工具，将正文中的数值作为结构化参数"
          "传入，验证逻辑链完整性。验证不通过则修正正文。")
    sys.exit(1)  # block:true时注入为提醒信息

sys.exit(0)
```

> **注意**：这个模式本质上是借用Hook的错误反馈机制来注入提示词——Hook输出非0时，
> Claude Code客户端会将stdout内容作为错误信息发回给LLM，LLM读取后执行相应操作。
> 文件实际上已经写入磁盘（PostToolUse触发），不会被回滚。

### 5.3 Hook不适合做的事：语义级验证（已迁移至MCP工具层）

以下验证在v1方案中被设计为Hook，现已迁移至第四章MCP工具层：


| 原Hook验证点        | 迁移后                               | 原因                                |
| --------------- | --------------------------------- | --------------------------------- |
| 样本量逻辑链完整性（数值比较） | MCP: validate_sample_size_logic   | 需要从自然语言中提取数值，正则不可靠；LLM以结构化参数调用更准确 |
| 对比试剂覆盖完整性       | MCP: validate_comparator_coverage | 需要理解靶标名称的语义对应关系                   |


### 5.4 Hook的配置方式

在项目的 `.claude/settings.json` 中配置：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "condition": {
          "filePatterns": ["*方案*", "*protocol*"]
        },
        "command": "python3 scripts/validate_qual_adaptation.py $FILE",
        "block": false
      },
      {
        "matcher": "Write",
        "condition": {
          "filePatterns": ["*样本量*"]
        },
        "command": "python3 scripts/remind_validate_sample_size.py $FILE",
        "block": true
      },
      {
        "matcher": "Write",
        "condition": {
          "filePatterns": ["*.md"]
        },
        "command": "python3 scripts/check_placeholder.py $FILE",
        "block": true
      }
    ]
  }
}
```

`block: true` 表示检查不通过时阻断并要求LLM修正，`block: false` 表示仅发出警告。

### 5.5 触发范围的精确控制

Hook的`condition.filePatterns`按文件名匹配。如果方案是单一大文件（所有章节写在一个.md里），则需要配合脚本内部判断：

```python
# 脚本开头：粗粒度关键词判断，不是目标内容直接跳过
content = open(sys.argv[1]).read()
if "样本量" not in content and "六（一）" not in content:
    sys.exit(0)
```

如果采用逐章分文件撰写（如 `04_样本量估算.md`），则可直接用文件名精确匹配，避免不必要的脚本启动开销。

## 六、法规知识库的结构化

### 6.1 当前问题

每次写方案都要重新解析法规全文，既耗时又容易遗漏细节。本次实践中，三份法规的22条量化要求是通过人工阅读后逐条提取的，这个过程本身就可以结构化沉淀。

### 6.2 建议的知识库格式

```markdown

- 发布机构: 国家药品监督管理局
- 施行日期: 2019

### 结构化约束

#### 硬性约束（不可违反）
| 编号 | 要求 | 量化指标 | 适用场景 |
|------|------|---------|---------|
| M-01 | 临床试验机构数量 | ≥3家，代表不同地区 | 所有方案 |
```

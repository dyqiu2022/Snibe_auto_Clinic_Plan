# 如何搭建可靠、可复用的 AI Agent 工作流

> Skills、MCP、规则检查 三大组件的实践指南
>
> 面向已经了解 AI Agent 基本概念，想知道**具体怎么搭一套生产级工作流**的同事。

---

## 一、一个工作流的"可靠性"从哪来？

一个工作流能不能投产，看三件事：

| 问题 | 组件 | 解决方案 |
|------|------|---------|
| AI 不知道该怎么做 | **Skills** | 把方法写成操作手册 |
| AI 有些事做不好、做不到 | **MCP** | 接第三方工具 |
| AI 可能跳步或犯错 | **规则检查** | 设自动检查点 |

---

## 二、Skills——把"怎么做"写成文件

### 2.1 核心思想

> **不要每次教 AI 怎么做，把方法写成文件，让它自己读。**
>
> 非常像我们平时给实习生写的 **SOP（标准操作流程）**，只不过 Skills 是写给 AI Agent 读的。

Skills 就是一份 `.md` 文件，用自然语言写清楚：一个任务的目标、步骤、规则、输入输出。

```
项目/.workbuddy/skills/
├── 周报生成/
│   └── SKILL.md
├── 报销整理/
│   └── SKILL.md
├── 方案撰写工作流/
│   ├── SKILL.md                  ← 主编排器
│   ├── pre-提取说明书信息/
│   │   └── SKILL.md
│   └── ...更多子 Skill
└── 合同审查/
    └── SKILL.md
```

### 2.2 一份好的 Skill 长什么样

拿项目中"提取说明书信息"这个 Skill 举例：

```markdown
---
title: "提取说明书信息"
summary: "从考核试剂说明书中提取结构化产品参数"
read_when:
  - 主编排器调度"提取说明书信息"步骤
  - 需要从 IFU 中提取产品参数时
---

## 输入
- 考核试剂说明书（IFU），PDF 或图片格式均可

## 处理步骤
1. 读取说明书全文
2. 逐一提取以下 23 项参数：
   - 产品名称、预期用途、样本类型、检测靶标
   - 方法学原理（定性/定量）、检测时间
   - 灵敏度、特异度、符合率、PPV、NPV
   - 保存条件、有效期、规格型号……
3. 对每一项参数，标注"从说明书第 X 页/节提取"
4. 提取不到的参数，标注"IFU 未提供"

## 输出格式
输出为 `产品信息提取.md`，表格形式：
| 序号 | 参数名 | 参数值 | 来源 |
|------|--------|--------|------|
| 1 | 产品名称 | 七项呼吸道病原体核酸检测试剂盒 | IFU 封面 |

## 规则（不可违反）
- 所有参数值必须逐字引用原文，不允许改写
- IFU 未提供的参数标注"未提供"，不允许编造
- 输出文件必须保存到项目文件夹
```

### 2.3 怎么写才可靠？有什么好处？

| 原则 | 说明 |
|------|------|
| **输入要明确** | AI 需要什么材料？缺什么就报错 |
| **步骤要细** | 每个步骤可独立执行、可验证 |
| **规则写死** | 什么能做、什么不能做、边界条件 |
| **输出有标准** | 结果是文件？表格？通知？格式要求？ |

写好之后，Skill 不绑定具体数据——换一批数据、换一个人、换一个项目，只要输入格式一样就能复用。

### 2.4 Skills 支持"套娃"（树状调用）

一个 Skill 可以调用另一个 Skill，形成树状结构：

| 层级 | 角色 | 说明 |
|------|------|------|
| **根 Skill** | 主编排器，只管流程调度 | 不干活，只安排谁干什么 |
| **树枝 Skill** | 负责一个子模块，可继续拆 | 管一个阶段，内部再拆子任务 |
| **叶子 Skill** | 做一件具体的事，不再往下拆 | 说明书信息提取、样本量计算 |

规则：根 Skill 只调度不干活，叶子 Skill 只干活不调度。发现一个 Skill 里事情太多，就拆成多个。

## 三、MCP——给 AI 装上"专用工具"

### 3.1 为什么要用 MCP？

AI 有大量事情做不好——不是它"不聪明"，而是它**天生不适合**：

| 任务 | 为什么 AI 自己做不行 |
|------|---------------------|
| 算样本量、跑统计 | 大模型本质是"猜下一个词"，不是算数 |
| 读写 .docx/.xlsx/.pdf | 无法直接操作二进制文件 |
| 查数据库、做实时检索 | 模型知识有截止日期 |
| 持久化记忆 | 每次对话相互独立，没有长期记忆 |
| 打开网页搜集信息 | 不知道网页上现在有什么 |
| 发企业微信/飞书/邮件 | 没有系统权限 |

> **MCP = 通用协议，让 AI 调用第三方工具干这些事。凡是模型做不好的，别让模型硬做。**

### 3.2 MCP 怎么工作

不管什么类型的工具，调用模式都一样：

```
AI 发现自己干不好某件事 → 通过 MCP 协议调用工具 → 工具执行 → 返回结果给 AI
```

整个过程**对使用者透明**——你只看到结果，不需要知道背后调了什么。

### 3.3 什么时候该加 MCP？

```
判断标准只有一条：这件事让 AI 自己硬做，你敢不敢验收？
不敢 → 加 MCP
敢   → 不用加
```

## 四、规则检查——给工作流装"质检员"

> 规则检查 = 在关键节点设置条件，不满足就停。WorkBuddy 里的规则检查**写在 SKILL.md 里**，是 Skills 的一部分（自然语言，不需要代码）。Claude Code 里规则检查通过 hooks 实现（写脚本，系统强制触发）。两种实现方式，同一个目的。
>
> 下面说的是**规则检查这件事本身**——怎么设计检查点、怎么写规则——不管用哪种工具来实现。

### 4.1 为什么需要规则检查

AI 工作流可能出这些问题：跳步（没做完第二步直接跳到第四步）、遗漏（忘了检查关键条款）、编数据（找不到数据时自己造一个）。规则检查就是在关键节点设条件，不满足就停。

### 4.2 检查规则的三种形态

| 形态 | 时机 | 示例 |
|------|------|------|
| **前置检查** | 执行前 | "Phase 3 没完成 → 不能进 Phase 4" |
| **后置验证** | 执行后 | "输出文件有占位符残留 → 修复" |
| **约束守卫** | 全流程 | "定性产品不能写定量指标 → 禁止" |

### 4.3 怎么写检查规则

直接在 SKILL.md 里写自然语言，不需要代码：

```markdown
## 全局硬约束（违反任一，流程停止）
1. 启动新章节前，确认前一步已标记 ✅
2. 所有引用数据必须标注来源
3. 输出文件不能包含原始模板占位符
```

规则要**可验证**（别写"检查数据质量"，写"缺失值比例 < 5%？"）、**有事可做**（别只报错，写"发现占位符 → 尝试修复 → 修不了再报错"）。

## 五、三步搭建一个工作流

```
Step 1: 梳理流程 → 拆成"输入 → 处理 → 输出"
Step 2: 写 Skill  → 一个流程 = 一个目录 = 一个 SKILL.md
Step 3: 加 MCP 和规则检查
        AI 做不好或做不到的  → 加 MCP
        不能出错的             → 加规则检查
```

> 举个具体例子：按这个三步法设计临床试验方案工作流
>
> Step 1 理清流程：读材料提取参数 → 分步写 23 个章节 → 拼装验证 → 导出 Word
> Step 2 写 Skills：主编排器 + 5 个提取器 + 23 个章节 Skill + 独立验证 Agent
> Step 3 加 MCP（样本量计算、文件读写）+ 规则检查（定性产品禁写定量术语、约束清单前置确认、全案完整性检查）

搭完之后，每次把材料扔进文件夹，说一句"开始写方案"，AI 自动走完全流程。你只管验收最终 Word。

## 六、参考案例：如果让你来设计这个工作流，你会怎么做？

> 以下案例面向**从事临床试验方案撰写的同事**，用大家熟悉的场景演示**如何从零设计**一套 Skills + MCP + 规则检查 工作流。
>
> 如果你不是这个领域的，可以直接跳到**第七节总结**，不影响理解。

前面的五步法（梳理流程→写 Skill→加 MCP 和规则检查）是通用的。这一节我们走一遍**真实的设计决策过程**——不是教你怎么用这个工作流，而是教你怎么把它设计出来。

**场景**：你的任务是设计一个工作流，让 AI 能自动写出一份 IVD 临床试验方案（23 个章节、25 条法规约束、含样本量计算、最终出 Word）。

你会怎么设计？

---

### 第一步：拆业务——先别想 AI，先理清人的流程

任何工作流设计，第一步都不是写代码或写 Skill，而是**把你脑子里的业务逻辑搬到纸面上**。

临床试验方案撰写这个任务，人是怎么做的？

```
输入：考核试剂说明书 + 对比试剂说明书 + 指导原则文件
  ↓
1. 先读材料 → 提取关键信息（产品参数、法规要求）
  ↓
2. 写各个章节（背景、设计、统计……）
    注意：有些章节可以同时写，有些必须等前面的写完
  ↓
3. 检查有没有写错（格式对不对、法规落实了没）
  ↓
4. 导出 Word
```

把这个流程画成图，你就知道了哪些步骤可以**并行**、哪些必须**串行**、哪些环节容易**出错**。

> **这是最关键的一步。这一步理清楚了，后面写 Skill、加 MCP、设检查规则都是填空式的。**

---

### 第二步：搭骨架——设计 Skill 树

流程理清楚后，你要决定：**哪些步骤写成独立的 Skill？谁来调度？**

你的决策逻辑应该是：

| 你的判断 | 设计决策 |
|---------|---------|
| 这个工作流有明确的阶段划分（预检→写作→验证→导出） | 需要一个**主编排器 Skill**，只负责调度，不负责具体干活 |
| 提取材料这件事有 5 种不同的材料要处理 | 每种材料写一个**独立的提取器 Skill**，让主编排器调用它们 |
| 有 23 个章节要写 | 每个章节写一个**独立的章节 Skill**，但一些简单的（封面、伦理、职责）可以合起来 |
| 验证需要"自己查 + 别人查"两轮 | 写一个**验证子 Agent Skill**，独立于写作流程 |

这样设计出来的 Skill 树长这样：

```
.workbuddy/skills/方案撰写工作流/
├── SKILL.md                      ← 主编排器（只调度，不干活）
├── pre-提取说明书信息/
│   └── SKILL.md
├── pre-提取指导原则约束/
│   └── SKILL.md
├── pre-提取专家共识/
│   └── SKILL.md
├── ch-03-背景资料/
│   └── SKILL.md
├── ch-05-试验设计/
│   └── SKILL.md
├── ch-06-统计学/
│   └── SKILL.md
├── ...其他章节/
└── v-验证/
    └── SKILL.md                  ← 独立验证子 Agent
```

**主编排器 Skill 大概长什么样？**

它几乎不写具体内容，只写流程：

```markdown
## 工作流步骤

1. 检查项目文件夹中是否有以下材料：
   - 考核试剂说明书（IFU）
   - 对比试剂说明书
   - 指导原则文件？
   缺材料 → 报错，不继续

2. **并行执行**以下提取任务：
   - 调用 "pre-提取说明书信息" Skill
   - 调用 "pre-提取指导原则约束" Skill
   - 调用 "pre-提取专家共识" Skill（有文件就做，没有就跳过）

3. 提取完成后，整理约束分发清单

4. 开始写作（串行+并行，见各章节依赖关系）

5. 执行验证

6. 询问用户是否导出 Word
```

> **设计的核心原则：主编排器只做调度决策，不做具体工作。具体工作由叶子 Skill 负责。**

---

### 第三步：装工具——判断哪些事 AI 做不好

Skill 树搭好后，看每个具体步骤：AI 自己做靠谱吗？不靠谱就加 MCP。

按 Section 三那句判断标准走：**你敢不敢验收 AI 硬做的结果？不敢就加。**

在这个案例里，需要 MCP 的环节：

| 环节 | AI 为什么做不好 | 加什么 |
|------|---------------|--------|
| 算样本量 | 大模型不擅长精确数学 | 样本量计算 Python 脚本 |
| 读写 .docx | 无法直接操作二进制文件 | 文件处理脚本 |
| 发审批通知 | AI 没有系统权限 | 企业微信/飞书接口 |
| 查实时入组数据 | 知识有截止日期 | 数据库查询接口 |

---

### 第四步：设关卡——哪些环节不能出错？

工作流能跑了，但 AI 可能会跳步、遗漏、或写出违规内容。这不是"态度问题"，是大模型固有缺陷。

你的决策逻辑：**这个环节出了错，后果严重吗？严重就加检查规则。** 规则写在相应 SKILL.md 里，自然语言即可：

| 检查时机 | 检查什么 | 在 SKILL.md 里这样写 |
|---------|---------|---------------------|
| 第三章启动前 | 约束分发清单生成了吗？ | 前置条件：确认 XX 文件已生成 ✅，未生成停下来 |
| 第五章写完后 | 定性产品有没有写定量方法？ | 后置验证：检索"Bland-Altman""ROC"等词，违规就删 |
| 全方案拼装后 | 章节齐全吗？产品名统一吗？数值自洽吗？ | 完整性 / 一致性 / 数值校验 |

> 检查规则阻断后：AI 立刻停下来 → 定位违规内容 → 修复 → 重新检查 → 通过才继续。

---

### 第五步：加保险——设计独立验证

第四步的检查全部是"AI 自己查自己"。自己查自己，查不出自己想不到的问题。所以加一层：**设计一个没参与写作的独立子 Agent，拿着 25 条法规约束逐条复核。** 有没落实的，打回去让主 Agent 修改。

就像你写完方案，再交给一个没参与的同事复核——他能发现你的盲点。

---

### 第六步：交出去

设计完成后，把整套东西交给使用者。他只需要说一句"开始写方案"，剩下的全自动。

| 指标 | 数值 |
|------|------|
| 你的投入 | 写 30 个 SKILL.md + 1 个 MCP 脚本 |
| 使用者的操作 | 一句话 |
| 每次复用成本 | 0（换材料即可） |

---

## 七、对比：Skills vs MCP vs 规则检查

> 三个组件各自解决什么问题？有什么区别？到底该用哪一个？

| 维度 | Skills（含规则检查） | MCP | 规则检查（Claude Code 用 hooks） |
|------|---------------------|-----|--------------------------------|
| **一句话定位** | 告诉 AI"怎么做"，顺带自检 | 给 AI"做不好、做不到"的能力 | 系统强制检查，不经过 AI |
| **本质是什么** | 一份 .md 操作手册，里面可以写规则段落 | 一个外部工具接口（Python 脚本） | 配置文件里的事件触发器 |
| **要写代码吗** | 不需要，自然语言就行 | 需要，写 Python 脚本 | 需要，写 Shell/Python 脚本 |

> WorkBuddy：规则检查是 Skills 的一部分（写在 SKILL.md 里的自然语言段落）。Claude Code：规则检查通过 hooks 实现（系统触发脚本，100% 强制执行）。概念统一，实现不同。

### 什么时候该用哪个？

```
干活的方法不明确           → 写 Skill
AI 自己做不靠谱、做不到    → 加 MCP（文件读写 / 精确计算 / 查数据库 / 发通知……任何 AI 做不好的事）
流程不可跳过或质量需保证    → 加规则检查
三者组合 = Skills 写流程 + MCP 给能力 + 规则检查保质量
```

## 八、总结

```
可复用的工作流 = Skills（方法） + MCP（工具） + 规则检查（质检）

做一次，用十次。
```

# IVD 临床试验方案主编排器

## 全局硬约束（所有 Phase 必须遵守）

违反以下任一约束，生成结果无效。

1. **执行每个 Phase 前，必须 `Read` 该 Phase 列出的 skill 文件全文。** 不得凭记忆、概述或 skill 文件名推断内容。
2. **执行每个章节前，必须 `Read` 对应的模板文件全文。** 不得仅靠 skill 中的字段映射表生成。
3. **样本量计算必须使用确定性工具**（MCP 或 `uv run scripts/sample_size_calc.py`），禁止 AI 手算。
4. **定性/定量闸门必须严格执行**：定性产品删 5 个定量评价指标、4 个定量统计方法、2 个定量样本量方法。`grep` 验证必须通过。
5. **法规约束不可静默忽略**：Phase 0.7 必须将每条 H- 编号的硬性约束显式分发到对应章节。Phase 2-4 每个子Agent的Prompt中必须携带分管约束列表（不携带全量约束）。Phase 7 按分发清单逐条核对，不得静默忽略任何未落实条款。
6. **流程检查不可跳过**：`流程检查.md` 是流程完整性的唯一可信记录。每个 Phase 启动前必须 `Read` 该文件并确认前置步骤已标记 ✅。每个 Phase 完成后必须追加一行标记（产物+行数+完成时间）。任何 ❌ 或 ⏳ 标记 → HALT。

---

## 职责

你是一个主编排器。你的任务是：用户把 IFU 和法规扔进项目文件夹 → 你自动读取所有文件并生成 `项目信息.md` → 按正确顺序调度每个章节的生成 → 拼装成完整方案 → 执行验证。

**用户只需要做一件事：** 把文件放进 `projects/<项目名>/` 的对应子目录里，其余全部自动处理。

### 0.7 约束分发清单（全局约束分发）

这是一个硬动作，不可跳过。法规约束必须显式分发，禁止靠子Agent自行从全量约束中盲挑。

#### 目标

将 `指导原则约束汇总.md` 中的全部 H- 硬性约束，逐条映射到对应章节，生成 `约束分发清单.md`。

#### 方法

1. 读取 `projects/<name>/指导原则约束汇总.md`
2. 提取每条 H- 约束的四要素：编号、条款要点（摘要，<=20字）、量化要求、影响章节
3. 按章节归并：

   | 影响章节中的关键词 | 归属章节 |
   |-------------------|---------|
   | 三、背景、流行病学、预期用途、产品描述、适用人群 | 03 背景资料 |
   | 三、五、亚型、基因型、流行株、季节 | 03 背景资料, 05 试验设计 |
   | 五、设计、入选标准、排除标准、样本类型、不一致确认、对比试剂 | 05 试验设计 |
   | 六、样本量、统计学、统计分析、Kappa、符合率、机构数 | 06 统计学考虑 |
   | 全方案、通用 | 编排器全局检查 |

   一条约束可能同时归属多个章节。

4. 写入 `projects/<name>/约束分发清单.md`
5. 统计覆盖：确认所有 H- 约束均已映射

## Phase 7：拼装与验证

### 7.1 拼装

### 7.2 验证检查（机械检查，编排器自己执行）

**V-01：占位符残留检查：** 搜索 `【` 占位符，必须返回零结果。

**V-03：产品名称一致性检查：** 封面、背景资料、对比表中的考核试剂名称一致。

**V-04：定性/定量内容完整性检查：** 定性产品 grep 禁忌词必须为零匹配。定量产品必须包含全部评价指标。

**V-05e 数值自洽：** 机构数 >= 法规最低、最终样本数 >= 统计估算数。

### 7.3 结尾验证 Agent（独立的约束落实核查）

### 7.5 导出 .docx（格式保留方案）

> **重要：** `scripts/ivd_export_docx.py` 已内置处理以下所有格式陷阱。执行导出时直接运行脚本，**不要**手动尝试修复 markdown 源码、**不要**额外做格式转换。脚本内部已包含：`**`标记清理、HTML表格解析（含colspan/rowspan）、段落合并、LaTeX→OMML方程、_Toc/BOM全局过滤、宋体标题、首行缩进、编码兼容。

完成 7.4 后，询问用户是否需要将 .md 方案导出为 .docx 格式。提供两种方式：

```
=== 方案已通过验证 ===
是否导出为 .docx？

选择：
  1. 使用 .docx 模板（推荐）— 保留模板格式、页眉页脚、表格边框
  2. 直接转换（无模板）— 从 .md 生成 .docx，标题粗体+表格三线表
  3. 否，仅保留 .md 格式
```

#### 7.5.1 方式一：模板填充（推荐）

**适用场景：** 已有现成的 .docx 模板文件（如公司固定格式的临床试验方案模板）。

**方法：** 使用 `scripts/ivd_fill_template.py <input.md> <template.docx> <output.docx>`

**原理：**
1. 加载模板 .docx → 保留全部原始样式（Heading 标题层级、Normal 正文格式、表格边框、页眉页脚）
2. 按章节标题匹配（"一、申办者信息""二、临床试验机构"…→"十九、参考文献"）定位填充位置
3. 封面区自动替换：方案编号、产品名称、版本日期、申办者信息
4. 表格内容替换：修订记录表、缩略语表、竞品信息表、对比表、四格表、签名页
5. 表格仅替换单元格文本，保留行数 + 格式（避免破坏模板表格布局）

**⚠️ 注意：**
- 模板中的旧内容（如上一个产品的名称、旧版本日期）会被自动覆盖
- 模板中的固定文本（页眉、页脚、研究者声明签名框）保留不动
- 如果模板的章节结构与 .md 不匹配，优先保留模板结构

**输出：** `output/【项目名】/临床试验方案_<产品全称>.docx`

#### 7.5.2 方式二：直接转换（无模板）

**适用场景：** 无现成 .docx 模板，需要从 .md 直接生成。

**方法：** 使用 `scripts/ivd_export_docx.py <input.md> <output.docx>`

**转换规则：**

| .md 内容 | Word 输出 | 实现方式 |
|----------|----------|---------|
| 标题 `# / ## / ###` | 粗体 + 字号分级（16/14/13pt），去 `**` | `add_heading_styled()` |
| 正文段落 | 宋体 12pt，连续行合并为一段 | `body_buffer` 累积 → 遇到空行/标题才 flush |
| `**粗体**` `*斜体*` | 对应 Word 粗体/斜体 | `markdown_to_inline()` 正则解析 |
| HTML `<table>` | Word 三线表（顶线+表头底线+底线），正确处理 colspan/rowspan | `parse_html_table()` + `add_table_from_md()` |
| 列表 `- ` / `1. ` | 独立段落 | 去标记后调用 `add_rich_paragraph()` |
| 空行 | 段落分隔（不生成空段落） | 跳过，触发 body_buffer flush |
| 引用 `> ` | 斜体段落 | 直接写入 |
| LaTeX 公式 `$...$` / `$$...$$` | 原生 OMML 方程（不是纯文本） | `add_omath_paragraph()`：LaTeX → MathML → OMML |

**已知的格式陷阱（全部已处理，重新生成时无需再调）：**

| 陷阱 | 表现 | 后果 | 修复方式 |
|------|------|------|---------|
| `**` 标记残留 | 标题中出现 `**一、申办者信息**` | 所有标题带脏 markdown 语法 | `add_heading_styled()` 开头 strip |
| HTML 表格不解析 | `<table>` 在 Word 里消失 | 方案里所有表格丢失 | `parse_html_table()` 正则提取 `<tr><td>` |
| colspan/rowspan 漏格 | 列联表（如四格表）缺少占位单元格 | 行列错位，数据对不上 | `parse_html_table()` 读取属性展开 cells，缺口补位 |
| `_Toc` 锚点残留 | `<a id="_Toc129858732"></a>` 出现在正文 | 方案正文出现乱码标签 | `convert()` 入口全局正则清理 + sed 预处理源文件 |
| 单行单段落 | 每个换行都变成独立 Word 段落 | 842 段落变独立段，间距呈 2 倍 | `body_buffer` 累积合并机制 |
| LaTeX 公式变纯文本 | `$a/(a+c) \times 100\%$` 直接显示源码 | 公式不可读，审评无法接受 | 全部替换为 OMML 方程（`add_omath_paragraph()`） |
| 公式检测正则崩溃 | `r'\\(?:frac\|sqrt...'` 里 `\f` 被当换页符 | 公式段落漏检测，变纯文本 | 改为 `chr(92) in text` 逐字符判断 |
| 中文编码报错 | `✅` 等 emoji 在 GBK 终端崩溃 | 脚本中断，导出失败 | 所有 print 用 ASCII 标记 `[OK]`/`[WARN]` |
| 标题字体黑体 | 所有标题渲染为黑体而非宋体 | 不符合中文方案排版规范 | `CN_FONT_BOLD` 改为 `"宋体"`（保留加粗属性） |
| 段落无首行缩进 | 正文顶格排版 | 不符合中文段落规范 | Normal 样式 `first_line_indent = Cm(0.74)` |
| Markdown pipe table | `| 缩略语 | 英文 | 中文 |` 格式脚本不解析 | 表格内容输出为 0 行 | 缩略语/参考文献等表格用 HTML `<table>` 格式而非 pipe |
| 签名框缺表 | 研究者声明签名区写成纯文本 | 无边框签名格，格式不规范 | 签名框用 `<table><tr><td>` 包裹，导出渲染为带框签名格 |
| LaTeX `\text`/`\begin{aligned}` | latex2mathml 生成非法 XML | OMML 转换崩溃，公式丢失 | 计算步骤拆为独立 `$$...$$` 块，避免复杂 LaTeX 环境 |

**输出：** `output/【项目名】/临床试验方案_<产品全称>.docx`

## 占位符解析规则（通用）

对于模板中出现的 `【XXX_描述】` 占位符，按以下优先级解析：

1. `项目信息.md` → 对应字段的值
2. `说明书/考核试剂/` → LLM 从 IFU 文本中提取
3. `说明书/对比试剂/` → LLM 从 IFU 文本中提取
4. 计算 → 从已解析的其他值通过公式推导
5. `指导原则/` → 从法规文件中提取
6. 询问用户 → 以上都无法解决时，明确提问
